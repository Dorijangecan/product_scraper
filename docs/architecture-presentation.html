What It Does

You upload a CSV or Excel file, choose the catalog-number column and select a manufacturer. The app creates a run, scrapes official and configured sources, normalizes technical attributes, grades each item and exports source-backed workbooks.

In practical terms, it replaces manual copy/paste work from manufacturer pages, datasheets and catalog PDFs. The operator gives it article numbers; the system builds a structured product dataset with source evidence, downloadable assets and a PDT-ready export path.

  1. Creates a local run in SQLite.
  2. Scrapes official manufacturer sources first.
  3. Uses configured fallbacks, discovery, rendered pages and document enrichment when the first result is weak.
  4. Normalizes technical attributes without a runtime model call.
  5. Grades each item with a quality gate.
  6. Exports a traced products.xlsx workbook and, on demand, a PDT workbook.

Input

CSV/XLSX with catalog numbers, selected manufacturer, optional customer documents and run options for images, PDFs, CAD, links or Excel output.

Processing

Official pages are tried first, then configured fallback URLs, search/discovery, rendered pages and document parsing when needed.

Output

Traceable Excel workbook, optional PDT workbook, downloaded documents/CAD/images, product links and debug logs.

The design goal is source-backed output. Values come from scraped pages, documents, customer uploads, parsed quantities or explicit deterministic rules. Unknown values are left missing and surfaced in diagnostics instead of guessed.

Important fast paths

  • Images-only runs skip Excel generation, broad fallbacks and PDF enrichment.
  • Links-only runs can produce discovered product URLs without image/document downloads.
  • Customer documents are moved into the run folder and treated as authoritative overrides for the run.

What happens inside a weak result

A weak scrape is not accepted immediately. The run manager can ask the deterministic fallback pipeline to try better candidates, use Playwright for JavaScript-heavy pages, parse downloaded datasheets for missing facts and run a final completeness check before the item is finalized.

Manufacturer Support

Built-in manufacturer profiles live in src/server/config/manufacturers.ts. Dedicated connector modules are registered in src/server/scrapers/index.ts.

The distinction is important: a dedicated connector is hand-written code for a difficult vendor website, while a config-driven profile uses generic scraping rules. This lets the tool support both complex vendors and simpler vendors without adding custom code for every new brand.

Understanding Engine

The normalization layer is deterministic. It keeps raw manufacturer attributes unchanged and adds a technical/normalized view when it can recognize a meaning.

This is the part that makes the scraper more than a downloader. Manufacturers write the same fact in many ways: different languages, different labels, decimal comma, units embedded in text, ranges, AC/DC notation and conditions such as "at 40 C". The understanding engine maps those forms into stable internal facts while preserving the original wording.

How to teach it more

  1. Add new property meaning to PROPERTY_ONTOLOGY.
  2. Add new units or sanity bounds in quantity.ts.
  3. Add material/color language terms in normalizer.ts.
  4. Add tests before relying on the new mapping.
Do not add scattered one-off regexes for every product unless there is no more general representation available.

No-hallucination rule

If the system cannot recognize a label or parse a value safely, it does not invent a normalized value. The raw attribute still remains in the workbook, and the unknown label can appear in diagnostics as a knowledge-base gap to improve later.

Quality Gate

quality-gate.ts decides whether an item is found, partial or failed.

flowchart LR
  R["ProductResult"] --> ID{"Identity
ok?"}
  ID -->|conflict| FAIL["failed"]
  ID -->|ok| REQ{"Required
data?"}
  REQ --> ELEC{"Device
rules?"}
  ELEC --> SCORE["Score
sources + docs"]
  SCORE --> TH{"Threshold?"}
  TH -->|yes| FOUND["found"]
  TH -->|no| PART["partial
fallback"]
  • Checks catalog identity conflicts.
  • Checks required sections, attributes and documents from manufacturer config.
  • Checks normalized fields and device-type-aware requirements.
  • Scores source/document/section depth.
  • Applies confidence caps for distributor or weaker rendered data.

The quality gate records attempted stages and missing fields in ProductResult.qualityGate and diagnostics. A weak result can trigger deeper fallbacks.

Found

Identity is confirmed, key requirements are present and score is above the configured threshold.

Partial

The page/product was likely found, but some important facts or documents are missing.

Failed

The system could not confirm the product, or the result conflicts with the requested catalog number.

For management reporting, this means the tool can separate "usable but incomplete" from "wrong or not found", which is safer than silently exporting low-confidence data as if it were complete.

PDT Export

PDT export is generated on demand from the run result and templates/master_pdt.xlsx.

The PDT exporter does not simply dump scraped columns into Excel. It classifies each product into a device type, routes it to the correct template sheets, resolves each PDT column from normalized facts and highlights required values that are still missing.

npm run audit:pdt validates the template, resolver coverage, profile mapping, ontology facts and benchmark fixtures.

PDT output is intentionally auditable. The generated workbook can show which values were filled, which are missing and which documents support the product record.

Storage And Output Layout

The SQLite database is data/scraper.db.

All run state is local. The database tracks status, per-row results, cached pages, learned endpoints and fields that have already been exhausted for a product. Output files are written next to the run so the workbook, documents and logs can be reviewed together.

TableHolds
runsRun status, counts, options and output paths.
run_itemsPer-row catalog result and serialized ProductResult.
page_cacheCached HTTP responses and fetched metadata.
learned_endpointsSuccessful official API/URL patterns.
exhausted_fieldsFields previously confirmed missing for a catalog number.

Output folders

The main workbook filename is <manufacturerShortName>.<inputName>.product-scrape-<runId>.xlsx.

The workbook includes product rows, attributes, technical attributes, alias dictionary, documents, sources, evidence, final audit and failure/diagnostic worksheets.

Why this layout matters

  • Runs are easy to archive or send internally because all artifacts are grouped under one run folder.
  • Downloaded documents and images can be traced back to the same run that generated the workbook.
  • Debug logs make it possible to explain why an item was partial or failed without rerunning immediately.

scripts/start-desktop.cjs sets PDT_AI_CLEANUP=0 when the variable is not already set. This keeps AI cleanup opt-in.

The user experience is a desktop app, but technically the UI and API are still standard web components running locally. That keeps development simple while avoiding the need for users to manually open a browser tab or manage server ports.

Run Lifecycle

flowchart LR
  START(["start"]) --> Q["queued"]
  Q --> R["running"]
  R --> C["completed"]
  R --> PING["pausing"]
  PING --> P["paused"]
  P -->|resume| Q
  R -->|cancel| X["cancelled"]
  R -->|error| F["failed"]
  C --> END(["end"])
  X --> END
  F --> END

On server start, resumeInterruptedRuns() picks up recent queued, running, pausing and cancelling runs. Runs older than the 5-minute interruption window are cancelled to avoid silently resuming stale work.

Pause/resume is designed for long batches. The current active row is retried on resume so the system does not persist a half-finished scrape as if it were complete.

Boundaries And Risks

The tool is built for public manufacturer data. It does not attempt login-only portals, CAPTCHA solving, paywall bypasses or network-control bypass behavior.

BoundaryReasonHow it is handled
Manufacturer site changesVendor websites can change layout or APIs without notice.Quality gate, diagnostics, fallback sources and connector tests reveal breakage.
Missing published dataSome manufacturers do not publish every PDT field.Missing fields are flagged instead of guessed.
JavaScript-heavy pagesSome data is hidden behind tabs, modals or dynamic API calls.Playwright rendering and network capture are used when needed.
Local PC differencesNode/native modules/Chromium can vary between machines.Installer and launcher manage Node runtime, npm packages and Playwright Chromium.
